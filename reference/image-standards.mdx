---
title: "Image Usage Standards"
description: "Rules for choosing screenshots, brand assets, demo imagery, and placeholders in CalBudget docs."
---

<Warning>
  Product docs must use real product screenshots for feature proof. Fake dashboard mockups, template assets, generated visuals, and marketing placeholders must not be presented as screenshots.
</Warning>

## The rule

Use the most honest image type available:

1. Real screenshot from the current production app.
2. Real screenshot from a seeded sandbox account.
3. Real screenshot from a local build that matches production behavior.
4. Clearly labeled illustration or placeholder when no real screenshot exists.

If an image did not come from a real CalBudget UI, do not call it a screenshot.

## Brand assets

These assets are safe for branding and navigation:

| Asset | Use |
| --- | --- |
| `public/brand/calbudget-icon*.png` | Favicons, app icons, small brand marks. |
| `public/brand/calbudget-logo.png` | CalBudget logo placement. |
| `public/brand/app-store-badge.svg` | App Store badge where mobile availability is documented. |
| `public/brand/calbudget-mark.svg` | Brand mark if supported by the target docs layout. |

Brand assets are not substitutes for product screenshots.

## BIZ vendor assets

These assets may be used for CalBudget Business vendor examples only:

| Asset group | Use |
| --- | --- |
| `public/images/biz-vendors/*.svg` | SaaS vendor examples in BIZ docs. |
| `public/images/svgs/{stripe,slack,figma,vercel,chatgpt,supabase,...}.svg` | BIZ vendor/integration examples when relevant. |

Do not use vendor logos to imply an active integration unless that integration is actually available in the product.

## Acceptable placeholder illustrations

`public/images/help/*.svg` assets are branded help-center illustrations. They are acceptable for:

1. Overview cards.
2. Empty-state style docs.
3. Help-center category thumbnails.

They should not be used to prove a workflow, setting, report, import result, or account state.

Use a caption such as:

> Illustration, not a product screenshot.

## Avoid or label as decorative

Do not use these as product proof:

| Asset or source | Reason |
| --- | --- |
| `public/images/dashboard/dashboard-mock-*.png` | Dashboard mockups, not trustworthy product screenshots. |
| `public/images/dashboard/analytics-report-demo.png` | Demo asset unless verified against current product. |
| Imported business template folders | Template/marketing imagery. |
| `public/images/bento-grid/*` | Template/marketing imagery. |
| `public/images/user/*` | Generic user imagery. |
| `public/images/team/*` | Generic team imagery. |
| `public/images/background/*` | Decorative backgrounds. |
| Placeholder logo asset sets | Placeholder logos. |
| `public/images/brand/brand-logo-*` | Placeholder brand-logo set. |
| `src/components/landing/MockViews.tsx` | Mock UI, not real product state. |
| `src/components/landing/demoData.ts` | Demo data, not customer/product truth. |
| `src/newmarketing/components/documentation/*` | Template documentation assets. |

If one of these appears in a work-in-progress docs branch, add a visible note:

> Placeholder image. Replace with a real CalBudget screenshot before release.

## Screenshot captions

Use captions that state what the user can actually see or do.

Good:

> The calendar shows daily running balances after scheduled income, bills, and recurring transactions are applied.

Avoid:

> CalBudget automatically predicts every future expense.

CalBudget forecasts from user-entered accounts, dated transactions, recurring generated rows, imports, and enabled beta sync data. It is not a bank prediction engine.

## Redaction standards

Before adding screenshots to docs:

1. Remove or blur emails, names, full account numbers, full card details, access tokens, support messages, and private notes.
2. Keep enough visible structure for the screenshot to teach the task.
3. Prefer seeded demo users over real customer accounts.
4. Do not show production admin screens in customer-facing docs.
5. Do not show private BIZ workspace data unless it is seeded demo data.

## Alt text standards

Alt text should describe the visible UI and the documentation purpose.

Good:

```mdx
<img
  src="/images/calbudget/business-hero.jpg"
  alt="CalBudget month calendar showing scheduled income, bills, and a forecast sidebar"
/>
```

Avoid:

```mdx
<img src="/images/calbudget/business-hero.jpg" alt="Screenshot" />
```

## Final publication checklist

Before a docs page ships:

1. Every feature walkthrough uses real screenshots or no screenshot.
2. Every non-real image is labeled as illustration, decorative, demo, or placeholder.
3. No template assets are presented as product UI.
4. No private production data is visible.
5. Captions do not overpromise behavior.
6. BIZ images appear only in BIZ docs unless clearly contextualized.
