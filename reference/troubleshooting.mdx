<Warning>
  Troubleshooting should start by confirming whether the user is on consumer CalBudget or CalBudget Business. The two product surfaces use separate routes, databases, and support expectations.
</Warning>

## Start with the surface

CalBudget has two separate product surfaces:

| Surface | Routes | Audience | Data boundary |
| --- | --- | --- | --- |
| Consumer CalBudget | `/calendar`, `/debt`, `/billing`, `/subscribe`, settings and support inside the app | Personal budgeting customers | `DATABASE_URL`, `src/db/schema.ts`, `src/db/migrations` |
| CalBudget Business / BIZ | `/business`, `/biz`, `/biz/calendar` | Verified businesses managing SaaS spend | `BIZ_DATABASE_URL`, `src/db/biz/schema.ts`, `src/db/biz/migrations` |

When troubleshooting, confirm which surface the user is on before giving steps. Consumer support should not mention BIZ admin-gated demo routes unless the user is explicitly using CalBudget Business.

## Login and account access

### The login page redirects to the homepage

CalBudget's authentication UI is launched from the public homepage. `/login` and `/register` intentionally redirect through the landing experience rather than behaving like traditional standalone forms.

Suggested user-facing copy:

> Go to `calbudget.com`, choose sign in, and enter the email you used for CalBudget.

### Password reset email did not arrive

Ask the user to:

1. Check the exact email they used for CalBudget.
2. Check spam, promotions, and filtered inbox folders.
3. Wait a few minutes before trying again.
4. Contact support if the reset email still does not arrive.

Internal note: password reset is handled by Neon Auth through CalBudget auth routes. Do not promise manual password changes in public docs.

### The app keeps asking the user to subscribe

Subscription access can come from Stripe, Apple, RevenueCat, or a temporary free-access grant. If a paying user is gated unexpectedly:

1. Ask which purchase path they used: web checkout, App Store, or another mobile flow.
2. Ask them to refresh the app and sign out/in once.
3. Send them to support with the email on the subscription receipt.

Do not ask users to paste full receipts, card numbers, or private payment details into docs comments or support screenshots.

## Calendar and transaction issues

### Balance looks wrong

Most balance issues come from one of these inputs:

| Check | Why it matters |
| --- | --- |
| Account starting balance | The forecast begins from the account's starting balance. |
| Transaction date | Balances are calendar-date driven. A transaction on the wrong day changes every later daily balance. |
| Income vs expense category | Income increases balances. Expense and uncategorized transactions reduce balances. |
| Cleared vs planned status | Cleared state affects reconciliation and confidence, but planned future transactions still appear in the forecast. |
| Recurring generated rows | A recurring series creates future occurrences that affect forecasted balances. |

Suggested user-facing copy:

> Open the affected account, check its starting balance, then review transactions on and before the first date where the balance looks off.

### A recurring transaction duplicated or appears on the wrong date

Ask the user whether they edited one occurrence, future occurrences, or the entire series. Recurring transactions share a recurrence group, and editing scope matters.

Recommended support steps:

1. Open one affected transaction.
2. Confirm the recurrence cadence, next date, and end date.
3. If duplicates exist, remove the unintended occurrence or edit the series scope.
4. Recreate the series only if the recurrence settings are badly wrong.

Use `biweekly` and `twice monthly` in public copy. Avoid "semi-monthly" unless the UI uses it in the current release.

### Imported transactions did not change the forecast as expected

Statement reconciliation asks users to match, create, or ignore rows:

| Decision | Result |
| --- | --- |
| Match | Existing transaction is marked cleared. |
| Create | A new cleared transaction is created. |
| Ignore | The statement row is skipped. |

If a row was ignored or matched to the wrong transaction, the user may need to re-upload the statement or manually edit the affected transaction.

## Reports and Navigator issues

### AI Financial Report is unavailable

The AI Financial Report is separate from CalBudget Navigator. It requires enough eligible past and future data and may have refresh limits.

Do not describe the AI Financial Report as Navigator chat. Do not promise that it can answer arbitrary questions.

### Navigator voice is not responding

Navigator voice depends on:

1. Navigator access.
2. The setting being enabled.
3. Browser speech support.
4. Microphone permission.
5. A supported interaction, such as holding `V`, tapping the mic, or typing a command.

Voice is for CalBudget commands such as creating transactions. It is not general dictation.

## Plaid bank sync beta

Plaid bank sync is admin-gated and may not appear for every user. If a user does not see Plaid:

1. Confirm whether the user's account is enabled for the beta.
2. Do not imply bank sync is the default CalBudget setup.
3. Offer manual accounts and statement import as the normal path.

If the user disconnects Plaid, existing CalBudget data remains in the app unless the user deletes it separately.

## Support escalation checklist

Collect:

1. User email.
2. Affected route or screen.
3. Account name, not full bank credentials.
4. Transaction date and amount, if relevant.
5. Browser/device.
6. Whether the issue is consumer CalBudget or CalBudget Business.
7. A real screenshot with private details redacted.

<Warning>
  Screenshots for support and docs must come from the real app or a seeded sandbox account. Do not use generated demo dashboards, landing-page mock views, or template marketing assets to demonstrate troubleshooting states.
</Warning>
