## Admin surface

The internal admin area lives under:

```txt
/quantum-nebula-9x
```

Access is guarded by admin authorization. Admin APIs use `requireAdmin()` and live under `/api/admin/*`.

Public docs may mention that support and billing are managed by CalBudget staff, but should not expose the hidden admin route path, admin screenshots, or operational implementation details.

## Admin audit expectations

`src/lib/admin-audit.ts` records best-effort audit rows. Failures are logged but do not block the admin action.

Audited action families include:

1. User create, update, delete.
2. User reset tools.
3. Billing reconciliation and selected subscription actions.
4. Feedback campaign create/update.
5. Feedback reward grants.
6. Newsletter sends.
7. Admin impersonation start/stop.

Known audit-log caveat: blog, changelog, and support mutations may not all record admin audit logs. Verify before writing compliance-facing docs.

## Impersonation documentation notes

Admin impersonation exists to help debug user-reported issues. Internal docs should explain:

1. Who is allowed to impersonate.
2. Which actions are appropriate during impersonation.
3. How to stop impersonation.
4. What audit events are written.
5. What support notes should be recorded after use.

Customer-facing docs should not encourage users to request impersonation as a normal support step.

## Admin screenshots

Admin screenshots are internal-only by default.

Before adding an admin screenshot to internal docs:

1. Use a seeded or local admin account where possible.
2. Redact user emails, names, support ticket text, invoices, revenue details, and identifiers.
3. Do not include production customer rows unless there is an explicit internal need.
4. Add a caption that says "Internal admin screenshot" or "Seeded admin screenshot."
5. Confirm the page is not included in public Mintlify navigation.

<Warning>
  Never use hidden admin screenshots in customer-facing docs. Never use fake/template screenshots to explain admin operations unless they are visibly labeled as placeholders.
</Warning>

## Public/private split

Use this split when turning internal admin content into published docs:

| Topic | Public docs | Internal docs |
| --- | --- | --- |
| Support tickets | How users contact support and see replies. | Admin queue, status handling, reply workflow. |
| Billing help | How users manage subscriptions or contact support. | Stripe invoice lookup, subscription state edits, reconciliation. |
| Changelog | Public release notes. | DB-backed changelog CRUD and publishing workflow. |
| Blog | Public articles. | Admin editor workflow and publishing caveats. |
| Feedback | User-facing prompts and rewards if relevant. | Campaign setup, activation, reward grant handling. |
| Admin audit | Usually omit. | What is audited, gaps, and review process. |
